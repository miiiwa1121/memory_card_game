## 選定理由

### Next.js 16 (App Router)

**選定理由**:
- **App Router**: ディレクトリベースのルーティングにより、ゲームごとに独立したルート（`/vegetable`, `/color` 等）を直感的に管理できる
- **SEO対応**: 各ゲームページに個別の `metadata` を設定でき、サーバーサイドでのメタタグ生成が標準サポートされている
- **静的書き出し (`output: "export"`)**: サーバーサイド処理が不要なゲームの特性上、完全な静的HTMLとして書き出すことでCDN配信に最適化
- **React Server Components**: サーバーコンポーネントがデフォルトとなり、クライアントバンドルサイズを最小化。インタラクションが必要なコンポーネントのみ `"use client"` で明示

**不採用とした候補**:
| 候補 | 不採用理由 |
|---|---|
| Vite + React | ファイルベースルーティングやSEOのための追加設定が必要 |
| Gatsby | ビルド速度とエコシステムの活発さでNext.jsが優位 |
| Astro | Reactコンポーネントの再利用性を重視したためNext.jsを選択 |

---

### TypeScript

**選定理由**:
- **型安全性**: カードデータ・ゲームロジックの型定義により、ランタイムエラーを事前に防止
- **IDE支援**: 補完・リファクタリング・ナビゲーションの精度向上
- **Next.js標準**: Next.jsがTypeScriptをファーストクラスでサポートしており、追加設定不要

---

### React 19

**選定理由**:
- **Next.js 16の要件**: Next.js 16が React 19を前提としている
- **Server Components**: RSCによりサーバー側で完結する処理をクライアントに送らずパフォーマンス向上
- **安定性**: メジャーバージョンとして十分に安定

---

### Vanilla CSS (CSS Modules)

**選定理由**:
- **スコープの安全性**: CSS Modulesによりクラス名が自動的にスコープ化され、ゲーム間のスタイル衝突を完全に防止
- **ゼロランタイム**: CSS-in-JSと異なり、ランタイムのJavaScriptオーバーヘッドがゼロ
- **依存ゼロ**: TailwindCSS等の追加ライブラリが不要で、`node_modules`の肥大化を防止
- **特殊ゲームの分離**: 音階・リズムゲーム等の独自デザインを持つゲームは専用CSSファイル（`sound-rhythm.css`）に完全分離し、他ゲームへの影響を防止

**不採用とした候補**:
| 候補 | 不採用理由 |
|---|---|
| TailwindCSS | ユーティリティクラスの学習コストとHTML肥大化。ゲームごとの独自デザインが多く、カスタムCSSの方が柔軟 |
| styled-components | SSR時のスタイル注入処理が複雑。静的書き出しとの相性に懸念 |
| Emotion | styled-componentsと同様のランタイムオーバーヘッド |

---

### Web Audio API

**選定理由**:
- **ブラウザネイティブ**: 追加ライブラリ不要で音の生成・再生が可能
- **低レイテンシー**: ゲーム体験に必須の即座な音声フィードバックを実現
- **音の動的生成**: 音階ゲームではピッチを動的に制御、リズムゲームではパターンをプログラムで生成
- **軽量**: 音声ファイルのダウンロードが不要で、バンドルサイズに影響しない

---

### Cloudflare Pages (Static Assets)

**選定理由**:
- **グローバルCDN**: 世界中のエッジサーバーから配信され、日本国内外のユーザーに高速なアクセスを提供
- **無料枠**: 個人プロジェクトに十分な無料枠（帯域無制限）
- **静的サイトとの親和性**: `output: "export"` で生成された `out/` ディレクトリをそのまま配信
- **GitHub連携**: GitHubへのプッシュで自動デプロイ
- **Wrangler CLI**: ローカルからの手動デプロイやプレビューデプロイも簡単

**不採用とした候補**:
| 候補 | 不採用理由 |
|---|---|
| Vercel | 無料枠の制限（商用利用不可）。Cloudflareの方がコスト面で有利 |
| Netlify | CDNのエッジロケーション数でCloudflareが優位 |
| AWS S3 + CloudFront | 設定の複雑さ。個人プロジェクトにはオーバースペック |

## ビルド・デプロイ構成

```
ソースコード (TypeScript / React / CSS Modules)
    ↓ next build (output: "export")
静的ファイル一式 (out/)
    ↓ wrangler deploy / GitHub連携
Cloudflare Pages (グローバルCDN)
```

### ビルド設定

| 項目 | 値 |
|---|---|
| ビルドコマンド | `npm run build` |
| 出力ディレクトリ | `out/` |
| 画像最適化 | 無効（`images.unoptimized: true`）※静的書き出しのため |
| バンドラー | Turbopack（開発時） |

## プロジェクト構成方針

### ディレクトリ構成

```
app/
├── layout.tsx          # 共通レイアウト（Header / Footer）
├── page.tsx            # ホーム画面（カードファンUI）
├── globals.css         # グローバルスタイル
├── vegetable/          # 野菜神経衰弱
├── sound/              # 音階神経衰弱
├── color/              # 色覚神経衰弱
├── language/           # 言語神経衰弱
├── morse/              # モールス神経衰弱
├── rhythm/             # リズム感神経衰弱
└── knowledge/          # 知識神経衰弱

components/
├── Header.tsx          # 共通ヘッダー
├── Header.module.css
├── Footer.tsx          # 共通フッター
├── Footer.module.css
├── Card.tsx            # 汎用カードコンポーネント
├── Card.module.css
├── GameBoard.tsx       # 汎用ゲームボード
└── GameBoard.module.css
```

### 設計原則

1. **共通化と分離のバランス**: 画像・色・テキスト型ゲームは共通コンポーネント（`Card.tsx`, `GameBoard.tsx`）で処理を統一。音声系ゲームは独自コンポーネントに分離
2. **ページ単位の独立性**: 各ゲームは独立したディレクトリに配置し、個別のメタデータ・スタイルを持つ
3. **外部依存の最小化**: React / Next.js 以外のランタイム依存ゼロを維持
